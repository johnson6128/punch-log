# ユースケース: 営業・稟議管理

プロジェクトは1つ以上の**営業案件**で構成される。稟議はプロジェクト全体ではなく**営業案件ごと**に実施し、連携IDと稟議No.も案件単位で管理する。案件のステータスが「実行中」に移行したタイミングで、その案件に対応する作業の予算執行が可能になる。

**営業案件のステータス遷移**: 提案中 → 稟議中 → 実行中 → 完了

---

## UC-SL01: プロジェクトと第1営業案件の作成

- **アクター**: 営業
- **概要**: 営業担当者が新規プロジェクトのコンテナと、最初の営業案件を同時に作成する。
- **基本フロー**:
  1. 営業がプロジェクト提案画面を開く。
  2. **プロジェクト情報**として以下を入力する:
     - プロジェクト名・概要
     - 担当営業
     - 備考（顧客情報・背景など）
  3. **第1営業案件**として以下を入力する:
     - 案件名（例: 初期導入 / フェーズ1）
     - 想定期間（開始予定日・終了予定日）
     - 想定売上目標
  4. 保存すると、システムがプロジェクト（ステータス「提案中」）と案件（ステータス「提案中」）を作成し、案件に**連携ID**（稟議システムとのリンク用の一意なID）を自動発行する。
- **例外フロー**:
  - アクティブなプロジェクト（提案中・実行中）と同一名称の場合、警告を表示するが登録は許容する。
- **事後条件**: プロジェクトと第1営業案件が「提案中」で作成される。予算の執行（工数・経費の登録）は案件の稟議承認後まで不可。
- **備考**: プロジェクト単位の詳細設定（予算・担当社員）はプロジェクト管理者が最初の案件承認後に行う（UC-PJ01）。

## UC-SL02: 営業案件の稟議申請

- **アクター**: 営業
- **概要**: 外部の稟議システムに提出した稟議の番号（稟議No.）を**営業案件に**記録し、案件ステータスを「稟議中」に移行する。
- **事前条件**: 対象営業案件が「提案中」ステータスであること。
- **基本フロー**:
  1. 営業が対象プロジェクトを選択し、さらに対象**営業案件**を選択する。
  2. 稟議No.（外部稟議システムで採番された番号）を入力する。
  3. 必要に応じて稟議提出日・稟議内容の概要を入力する。
  4. 「稟議申請」操作を実行すると、案件ステータスが「稟議中」に更新される。
- **例外フロー**:
  - 「提案中」以外のステータスの案件に対してこの操作を行おうとした場合、エラーを表示する。
- **事後条件**: 営業案件に稟議No.が記録され、案件ステータスが「稟議中」になる。
- **備考**: 連携IDは案件作成時（UC-SL01 / UC-SL04）に自動発行済み。稟議No.は外部システム側の採番であり、本システムでは記録・表示のみ行う。1つのプロジェクト内で複数の案件が異なるタイミングで「稟議中」になることがある。

---

## UC-SL03: 営業案件の稟議承認登録

- **アクター**: 営業
- **概要**: 外部稟議システムで承認が下りたことを本システムに反映し、**営業案件**のステータスを「実行中」に移行して予算執行を可能にする。
- **事前条件**: 対象営業案件が「稟議中」ステータスであること。
- **基本フロー**:
  1. 営業が対象プロジェクトを選択し、さらに対象**営業案件**を選択する。
  2. 「稟議承認済み」操作を実行し、稟議承認日を入力する。
  3. システムが案件ステータスを「実行中」に更新する。
  4. プロジェクト内で初めて「実行中」になった案件の場合、プロジェクト自体のステータスも「実行中」に自動移行する。
  5. プロジェクトが「実行中」になったタイミングでプロジェクト管理者が詳細設定を行う（UC-PJ01）。
- **例外フロー**:
  - 「稟議中」以外のステータスの案件に対してこの操作を行おうとした場合、エラーを表示する。
- **事後条件**: 案件ステータスが「実行中」となり、当該案件に対応する工数・経費の登録が可能になる。
- **備考**: 稟議が否決された場合は案件ステータスを「提案中」に戻し、内容修正のうえ再申請または取り下げができる。取り下げの場合は案件を削除またはクローズする。

## UC-SL04: 営業案件の追加

- **アクター**: 営業
- **概要**: 既存プロジェクトに新しい営業案件（フェーズ・クォーターなど）を追加する。追加した案件はそれぞれ独立した稟議フローを経る。
- **事前条件**: 対象プロジェクトがクローズ済みでないこと。
- **基本フロー**:
  1. 営業が対象プロジェクトを選択し、「営業案件の追加」を操作する。
  2. 以下の情報を入力して保存する:
     - 案件名（例: フェーズ2 / 2026年Q2 / 追加開発）
     - 対象期間（開始日・終了日）
     - 売上目標金額
     - 備考
  3. システムが案件をプロジェクトに紐付け、ステータス「提案中」で保存し、**連携ID**を自動発行する。
  4. プロジェクト全体の売上目標が各案件の合計として更新される。
- **例外フロー**:
  - 案件の期間がプロジェクトの期間外の場合、警告を表示する（保存は許容する）。
- **事後条件**: 新しい案件が「提案中」で追加される。この案件は UC-SL02 → UC-SL03 の稟議フローを独立して経る。
- **備考**: プロジェクトが既に「実行中」であっても新しい案件を追加できる（追加スコープの稟議取得が目的）。案件を分割しない場合、プロジェクト全体を1つの案件として扱う（暗黙の一括案件）。

---

## UC-SL05: 営業案件ごとの売上登録

- **アクター**: 営業
- **概要**: 実績売上（請求額・入金額）を営業案件単位で登録する。
- **事前条件**: 対象営業案件が「実行中」または「完了」ステータスであること。
- **基本フロー**:
  1. 営業が対象プロジェクト・営業案件を選択する。
  2. 以下を入力して保存する:
     - 売上金額
     - 計上日
     - 摘要（請求書番号・入金区分など）
  3. 複数回に分けて登録可能（分割請求・マイルストーン請求に対応）。
  4. システムが案件の売上合計・プロジェクト全体の売上合計を更新する。
- **例外フロー**:
  - クローズ済みプロジェクトには売上を登録できない。
  - 「提案中」または「稟議中」の案件には売上を登録できない。
- **事後条件**: 登録した売上がプロジェクト収支レポート（UC-PJ09）に反映される。
